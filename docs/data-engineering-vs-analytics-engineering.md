# 이 실습에서 데이터 엔지니어링과 분석 엔지니어링을 어떻게 나누는가

이 실습의 핵심은 **“로그가 usable data가 되는 경로”**를 보는 것입니다.  
그래서 README 본문은 데이터 엔지니어링 경로에 집중하고, 분석 엔지니어링 관련 내용은 이 문서로 분리합니다.

## 한눈에 보는 역할 구분

| 구분 | 이 실습에서 다루는 핵심 질문 | 대표 산출물 |
|---|---|---|
| 데이터 엔지니어링 | 로그를 어디에 어떻게 쌓고, 적재하고, 모델링 가능한 형태로 만들까? | JSONL spool, MinIO object, DuckDB table, dbt model |
| 분석 엔지니어링 | 어떤 이벤트/메트릭 정의가 의사결정에 적합한가? 어떻게 지표를 해석할까? | metric definition, semantic layer, experiment analysis |

## README 본문에 포함되는 것

- 브라우저 beacon 이벤트 발생
- Flask 수집 endpoint
- 로컬 raw spool
- MinIO raw 업로드
- DuckDB 적재
- dbt staging/mart 모델
- 정적 HTML 리포트

이 단계는 모두 “데이터를 usable 상태로 만드는 공급망”에 해당합니다.

## `docs/`로 분리하는 것

다음 주제는 중요하지만, 메인 실습 경로를 복잡하게 만들기 때문에 별도 문서에서 다룹니다.

1. 이벤트 명세를 어떤 조직 규칙으로 설계할지
2. 지표 정의 시 분모/분자/세그먼트를 어떻게 관리할지
3. A/B 테스트 실험군/대조군을 어떻게 구분할지
4. 장기적으로 semantic layer나 metric layer를 어떻게 둘지

## 왜 이렇게 나누는가

초심자가 처음부터 모든 것을 한 번에 배우려 하면 보통 다음 문제가 생깁니다.

- 도구 사용법은 따라 했지만, 파이프라인의 경계를 설명하지 못함
- 로그 수집과 지표 해석을 한꺼번에 섞어 배워서 역할 분리가 흐려짐
- README가 너무 길어져서 “지금 뭘 검증하는 단계인지” 놓침

따라서 이 저장소는 다음 원칙을 따릅니다.

1. **README는 데이터 흐름에 집중한다.**
2. **이벤트/메트릭/A-B 실험은 보조 읽을거리로 분리한다.**
3. **최종 리포트는 분석 과제가 아니라 파이프라인 검증 결과물로 본다.**

## 이 실습을 마친 뒤 확장할 수 있는 질문

- `answer_submitted`를 기준으로 문제 난이도를 추정하려면 어떤 추가 필드가 필요할까?
- 세션 단위 retention이나 funnel을 보려면 어떤 이벤트가 더 필요할까?
- A/B 테스트를 위해 variant 할당 정보는 언제, 어디에 기록해야 할까?
- dbt mart를 metric-ready table로 확장하려면 어떤 dimension이 필요할까?

## 추천 학습 순서

1. 먼저 README의 end-to-end 실습을 끝낸다.
2. raw → storage → warehouse → model → report의 흐름을 설명할 수 있는지 확인한다.
3. 그 다음에 이벤트 설계/메트릭/A-B 테스트 문서를 읽는다.

이 순서를 지키면 “도구 나열”이 아니라 “데이터 제품화 경로”로 이해하기 쉬워집니다.
